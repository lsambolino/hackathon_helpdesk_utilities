"The Brake" вҖ” escalation rules, not vibes

Enforced by the coordinator's permission callback (not just the prompt)

  • confidence < 0.75 вҶ’ escalate low_confidence
  • amount > вӮ¬500 вҶ’ escalate high_amount
  • customer.vulnerable = 1 вҶ’ escalate vulnerable_customer
  • prompt-injection / DAN / authority spoof / chained instructions вҶ’ escalate adversarial
  • specialist returns is_error twice вҶ’ escalate specialist_failure
  • B2B contract dispute (IVA, voltura, SLA) вҶ’ escalate policy_exception

Vulnerable-customer status is enforced at the permission layer, so even a confident agent cannot auto-resolve those tickets вҖ” by construction.

Forecast & prevention вҖ” by zone

The coordinator surfaces a spike-risk panel on the dashboard: per-zone ticket volume, recent-vs-prior growth, and a risk classification.

Heuristic today (rule-based threshold over the last 30 days). Plug-in design: production version swaps in Prophet / ARIMA over a year of complaint volume вҖ” same API contract, same UI panel.

Why rule-based now

  • Disclosed: not an ML model.
  • Decisive: explainable per-zone score.
  • Drop-in: forecast_zones() is a single function, swap implementation behind it.

End-user chatbot вҖ” escalation by design

The customer-facing specialist runs the same policy as the back-office coordinator: it tries to resolve via KB and account context, and opens a ticket only when warranted.
